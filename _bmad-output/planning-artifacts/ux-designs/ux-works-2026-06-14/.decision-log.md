## 2026-06-14 — Discovery decisions (full set)

### D1 — Form-factors & primary design surfaces

Multi-surface, three surface classes:

1. **Web shell** — Blazor + Fluent UI v5 + FrontComposer. **Desktop-first, responsive** down to
   tablet; phone = read + simple-advance. The coordination console for internal users,
   coordinators, tenant admins, auditors, builders. *Primary surface that needs real design.*
2. **Email-as-UI** — the external-party surface. **Mobile-first, no login, one tap.** *Second
   surface that needs real (bespoke) design — it lives outside the shell.*
3. **MCP + Chatbot** — agent / conversational surface (capture + advance via MCP tools or a single
   chatbot sentence; projections exposed as tenant-scoped Markdown resources). **Treated as a
   contract surface**, not visual design — but it carries the "answer in my words" NL escape hatch.

*Rationale:* PRD says "channel and executor are orthogonal"; creation is omnichronal. The two
surfaces with genuine visual/interaction design weight are the shell and email. MCP/chatbot is a
contract the spines reference, not a screen set.

### D2 — Information architecture (web shell)

Shell nav (left rail via `<FrontComposerShell>`):

| Surface | Reached from | Purpose |
|---|---|---|
| **What's next** | App open (doer landing) · `g n` | My queue: assigned-to-me (push) + claimable (pull), ordered Priority → Due |
| **Work** | Rail · `g w` | Browse/search work items; the Work Tree + roll-up "one number" |
| **Work Item detail** | Queue/tree row | Obligation · Burn-Down · Schedule · Executor · Await · children · unified history |
| **Capture** | Global (`c` / palette) | "Capture in seconds" quick-create; mirrors email one-liner & chatbot sentence |
| **Admin** | Rail (admins) | Tenant policy: escalation ladders, authority levels, spend caps *(Themes 4/5)* |
| **Audit** | Rail (auditors) | Query the signed non-repudiation record *(Theme 6)* |

*Surface closure:* every PRD-implied need maps to a surface, and every surface has a Key Flow that
lands there (UJ-1..4 + handoff). Admin/Audit are deferred-theme surfaces — specified for
seam-shaping, marked *(Theme N)*.

### D3 — The signature: progress is a fact, not a status flag

The **Burn-Down meter** is the load-bearing visual. Horizontal bar: **Done** (filled,
progress-green) over a neutral **track** = Remaining; the **headline number = Remaining** in
tabular figures. `Done = Remaining is 0` flips status to Completed and fills the bar. Heterogeneous
Units are **never implicitly summed** — per-Unit subtotals shown side by side with the Unit label.
Live-updates via SignalR as progress is reported. This is the one component every surface carries.

### D4 — "One number" roll-up

A parent shows **own Remaining** and **subtree-rolled Remaining** as two figures (`metric-hero`
for the rolled number). Per-Unit subtotals preserved up the tree (no cross-Unit conversion). The
Work Tree node carries a mini burn-down + status pill + Party chip.

### D5 — "Everything is a Party" → one uniform executor treatment

A single **Party chip** for *any* executor (system/AI, internal user, external party): avatar +
name + a tiny **channel glyph** (MCP/CLI/Chatbot/Email) + a tiny **kind glyph**
(bot/person/external). **Same visual treatment regardless of kind — no color-coding by executor
kind** (this is the core principle; branching the UI on kind is an explicit anti-pattern).
**AuthorityLevel** {Read, Contribute, Coordinate, Administer} shown as an escalating monochrome
badge (carried, not enforced in v1). **Handoff/assign = one symmetric action** ("Hand off…") with
identical affordance whether the target is a bot, a colleague, or an emailed stranger.

### D6 — Push & pull coexist

Queue distinguishes **assigned-to-me** (push) from **claimable** (pull). **Claim** is optimistic;
**"single claim wins"** → if you lose the race the row updates to claimed-by-other with a plain
message ("Someone else got there first."). No modal confirm — Works trusts the user.

### D7 — Unified history (events + comments = one stream)

The activity timeline merges the Domain Event stream and the comment stream — "two views of one
history." Each entry renders the **Raw Act** verbatim (acting Party + timestamp + payload),
past-tense verb. Filterable by act type. No separate "audit log" vs "comments" surfaces.

### D8 — Email-as-UI (the headline external UX)

The email **is** the UI. AI embeds a **constrained-safe action set** — each button = one valid
domain act, **single-use, expiring, signed** ("one tap = work progressed") — plus a **"None of
these — answer in my words"** free-text escape hatch that maps NL onto the item's valid action
space. The answer-contract does triple duty: UX accelerator, input validator, prompt-injection
boundary (**NL is data, never instructions**). Must render in real email clients: table layout,
inline styles, system fonts, ≥44px tap targets, plain-text fallback. *(Theme 3.)*

### D9 — Visual foundation: Fluent UI v5, brand-layer delta only

Inherit **Fluent UI v5 (RC)** wholesale (neutrals, elevation, shapes, Segoe UI Variable type,
WCAG-AA defaults). DESIGN.md specifies only deltas:
- **Brand accent** `#5B5FC7` (Fluent "blurple" — Fluent-native *and* signals the AI-native /
  collaboration / "digital coworkers" identity). `[TASTE]` — alternatives rendered in the
  color-theme exploration; swappable in one place.
- **Burn-down semantics:** `burndown-done` `#107C10` (Fluent success green) over `burndown-track`
  (Fluent neutral). Cost meter (Theme 5) = warm gold `#C19C00`, a *second* burn-down, distinct
  from effort.
- **Status vocabulary** (lifecycle, grouped by phase): created grey · assigned blue · queued teal
  (pool) · in-progress blurple (live) · suspended amber (waiting) · completed green · cancelled
  muted-grey · rejected red · expired desaturated-red. Status is never the *only* progress signal
  (see D3).
- **Typography delta:** a `metric` role — **tabular figures** for every burn-down / roll-up number
  so digits don't jitter as they update live; `metric-hero` for the "one number."
- Everything else (spacing, elevation, corner radii, component visuals) inherits Fluent v5.
  Density follows FrontComposer's **spec-locked tiers** (≤1 field inline · 2–4 compact · ≥5 full
  page) — not a hand-authored scale.

### D10 — Composition strategy (design leverage on FrontComposer)

Most surfaces are **generated** from Works' `[Projection]`/`[Command]` types. Bespoke design effort
concentrates on: the **Burn-Down meter** & **roll-up** (L3 field slots), the **What's-next queue**
(FC-TBL DataGrid + L2 projection template), the **Work Item detail** (L2 template), the **history
timeline** (L2/L3), and the **email-as-UI** (fully bespoke, outside the shell). Admin/auditor
surfaces are generated forms/queries. EXPERIENCE.md carries a **Composition Map** (invented
section) so downstream dev knows which level each surface uses.

### D11 — Voice & tone

Factual, terse, no cheerleading; mirror PRD verbatim phrases ("what's next?", "one number", "one
tap = work progressed", "none of these — answer in my words", "capture in seconds"). Past-tense
event verbs in history. Errors are plain, ProblemDetails-derived. Same tone to every audience
(builder, coordinator, external party) — the system talks to everyone the same way.

### D12 — Accessibility & tenancy floor

Inherit the ecosystem **a11y/visual specimen gate** (Playwright `test:a11y`) → WCAG 2.2 AA (Fluent
v5 AA by default; brand deltas contrast-verified). Burn-down **never conveyed by color alone**
(number + bar + label). SignalR updates via `aria-live`. **Tenant isolation reflected in UI** —
cross-tenant data never rendered (query-side filtering); permission-denied surfaces are *hidden*,
not blocked screens. Email links labeled with the act they perform; plain-text fallback.

### D13 — Key flows (named protagonists) — map to PRD UJs

- **UJ-1** Builder wires Works in — *Sam*, via Aspire host + generated forms.
- **UJ-2** AI agent burns down work on the uniform surface — *Atlas* (a digital coworker) over MCP.
- **UJ-3** Spawn → suspend → resume saga — *Dana*, coordinator; parent auto-resumes on child done.
- **UJ-4** Email capture + external one-tap — *Mary* (PRD persona) + an emailed supplier, no login.
- **+Handoff symmetry** — folded into UJ-2/UJ-3 (the "one operation" climax).

### Out-of-scope guards (from PRD non-goals)

No BPMN/process-diagram authoring; no task-content storage UI; no AI in the system of record
(interpretations are projections); **don't over-fit v1 to deferred themes** — Admin/Audit/Cost
surfaces are specified thin, for seam-shaping only.
