## 3. What Went Well

1. **A pure, caller-fed design pattern that compounded.** Story 3.1's `WorkTreeAttachmentGuard` (validate from caller-supplied facts, zero I/O) became the reusable foundation that 3.2 (`SpawnChild`) consumed unchanged. The same "pure mechanical translator" shape from 3.5 (`ChildCompletionResumeTranslator`) was mirrored by 3.6 (`TerminalCascadeTranslator`) — added additively, **with no fitness-guard relaxation**.
2. **"Verify, don't re-implement" reconciliation.** Stories 3.4 and 3.6 both opened with a Task 1 reconciliation that *proved* prior stories already satisfied much of the new ACs, avoiding speculative scope. Story 3.6 needed **no production code change** to the aggregate or lifecycle — existing terminal idempotency + roll-up zeroing already made cascade idempotent. This honored the SM-C1/SM-C2 counter-metrics (don't grow the kernel; don't over-fit deferred themes).
3. **Lessons propagated across stories.** Story 3.3's CRITICAL (a property test using one fixed tree while claiming to generate trees) was directly applied in 3.4, which was explicitly directed to *genuinely generate* varied unit combinations and the degraded case. 3.5/3.6 property/translator tests followed suit.
4. **Defense-in-depth on tenant safety and unit safety.** Tenant equality is re-checked at every roll-up traversal hop and again in the cascade translator; unit-mismatch is guarded on the write side (command rejection) *and* mirrored read-side (fail-closed "refuse + retain + flag", never coerce).
5. **Clean architectural boundaries held under pressure.** Even when a "logging" requirement (3.4 AC#5) collided with a Contracts-only projection, the team produced `RollUpProjectionDiagnostic` as pure metadata-only data for a future adapter rather than injecting `ILogger` into the kernel.

## 4. What Was Challenging

1. **Test-count bookkeeping drift — the dominant recurring finding.** Dev Agent Record counts drifted from `tests/test-summary.md` in **3.2** (375→386), **3.3**, **3.4** (332/27→335/28), **3.5** (stale debug log), and **3.6** (dev claim 477 vs review 481). Each was caught at review (Medium/Low) and corrected. The numbers were never *wrong in the suite* — only mis-recorded in the narrative.
2. **Dev pass under-covers; a QA gap-filling pass is consistently required.** Every story needed a follow-up QA pass to reach approval: 3.1 (+4), 3.2 (+11), 3.4 (+3), 3.5 (**+22**). The first implementation pass repeatedly left coverage gaps (notably property-generator breadth and boundary cases).
3. **`dotnet test` is unusable in the sandbox.** Across all six stories, Microsoft.Testing.Platform named-pipe permissions block solution-level `dotnet test`; the reliable path is restore → build → run the xUnit v3 binaries directly under `bin/Release/net10.0/`. This friction is carried from Epic 2 and will directly impede Epic 4's runtime/integration-heavy stories (4.5, 4.6).
4. **Constants that cannot be physically shared across the purity boundary.** `SpawnChild.MaxDepth = 32` (Contracts) duplicates `WorkTreeDepthPolicy.DefaultMaxDepth = 32` (Server) because Contracts cannot reference Server. The mitigation is a doc-comment sync requirement only — a standing drift risk.
5. **EventStore projection-model mismatch surfaced but unresolved.** Story 3.3 noted the live EventStore source exposes a stateless full-replay `/project` seam via `IDomainProjectionHandler`, while the architecture/`docs/eventstore-api-surface-constraints.md` describe checkpoint-per-aggregate pausable rebuild. The pure projection logic was built first and runtime adapter wiring deferred — but the two models must be reconciled before any Epic 4 projection adapter lands.

## 5. Key Insights

- The pure-core / deferred-runtime split is paying for itself: each story shipped a fully tested domain behavior whose runtime counterpart is a separate, named, deferred seam. The risk is that **all** of that deferred runtime work is now concentrated in Epic 4 — especially Story 4.6.
- Idempotency designed once (terminal no-op in the lifecycle table) removed the need for a cascade-specific dedup mechanism entirely. Good invariants reduce later code.
- The single source of truth discipline works: `docs/lifecycle-transition-matrix.md` is mirrored 1:1 by `WorkItemLifecycle.cs`, and 3.6 explicitly refused to add a second cascade transition table.
