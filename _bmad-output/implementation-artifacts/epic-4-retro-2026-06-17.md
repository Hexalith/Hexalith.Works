## 3. What Went Well

1. **"Verify, don't re-implement" reached its peak.** Stories 4.2 and 4.3 shipped with **zero production-code changes** — each opened with a Task 1 reconciliation that *proved* the uniform assign/requeue/claim surface and the expected-version substrate already satisfied the new ACs, then closed the work with behavioral-proof tests, a fitness guard, and a decision-record note. 4.1 was near-zero (validation hardening only). This is the SM-C1/SM-C2 discipline ("don't grow the kernel; don't over-fit") working exactly as designed.

2. **The catalog stayed frozen at 36 for an entire runtime epic.** Single-claim-wins reused the existing `WorkItemTransitionRejected(InProgress, "Claim")` instead of inventing `ClaimRejected`/`ConcurrencyRejected` (DC1). The what's-next read model (`WhatsNextItem`) and every reminder/checkpoint record are plain STJ types, not `[PolymorphicSerialization]` catalog entries. Runtime behavior was added without expanding the durable wire contract at all.

3. **Deterministic concurrency and recovery proofs, not timing races.** 4.3 proved single-claim-wins by handling two claims against the same observed version (winner advances state, loser re-handles to a rejection) plus an FsCheck convergence property (K claims, any order → exactly one accepted) — no threads, no sleeps. 4.6 proved reminder reconciliation and mid-cascade restart replay deterministically with fakes + injected `TimeProvider`, keeping the live Tier-3 lane as a *separate* gated proof rather than the primary coverage.

4. **The QA gap-filling pass became routine, not remedial.** Every story ran `bmad-qa-generate-e2e-tests` as an expected step (4.1 +12, 4.2 +2, 4.3 +1, 4.4 +9) — directly applying Epic 3's Action Item 2. Gaps were closed *before* review, and the gaps that showed up were exactly where Epic 3 predicted: ordering edge cases, cross-tenant negatives, and "is this a fresh raw act or a NoOp?" assertions.

5. **Kernel purity survived the hardest test — the runtime layer.** Date resumes go through a Dapr `DateReminderActor` with deterministic reminder names; cascade recovery uses bounded checkpoint records and a re-readable descendant source — all in `src/Hexalith.Works`, with the aggregate still clock-free and infrastructure-free. The pure `TerminalCascadeTranslator`/`ChildCompletionResumeTranslator` were consumed unchanged; every domain decision still round-trips through `Handle`.

6. **The EventStore adapter-edge pattern was found and documented cleanly.** 4.5 discovered that `AssemblyScanner` only finds concrete `EventStoreAggregate<TState>` subclasses, so the host provides a thin `WorkItemEventStoreAggregate` (14 `Handle` wrappers delegating verbatim to the pure kernel) decorated `[EventStoreDomain("work")]`. The kernel was not touched; the boundary is fitness-asserted.

## 4. What Was Challenging

1. **A CRITICAL false-completion claim in the load-bearing story.** Story 4.6's first pass marked Tasks 6 & 9 done while the gated Tier-3 Aspire reminder-recovery lane (`WorksReminderRecoveryPipelineSmokeTests`) was **never authored** — the deterministic logic was proven, but the live restart proof did not exist. Review caught it, the tasks were reopened, and the lane was written (park on `DateReached` → restart AppHost against the same Redis → reissue via the production `DateResume.BuildSubmission` factory → prove exactly one `WorkItemResumed`, idempotent). This is the single most important finding of the epic: in the one story that concentrated all deferred runtime work, the most complex acceptance proof was the one quietly skipped.

2. **Test-count bookkeeping drift — still recurring, though improving.** Epic 3's #1 finding persisted: 4.1 carried stale verification counts (recorded 516, never mentioned the QA pass that reached 528) and 4.3 mis-reported 448/561/+12 vs the actual 449/562/+13 (review findings M1/M2). The good news: 4.2, 4.4, and 4.5 were clean and reviewer-verified to match exactly. Epic 3 Action Item 1 is **partially** honored — the discipline is better but not yet zero-drift.

3. **The substrate cannot do tenant-wide / domain-wide discovery yet.** Both 4.5 and 4.6 hit the same wall: EventStore's `StreamsController` rejects a null `AggregateId` (per-aggregate stream reads only), even though `StreamReadRequest.AggregateId` is contract-optional. So reminder reconciliation cannot auto-discover all pending `DateReached` awaits across a tenant, and cross-aggregate rolled-remaining cannot be assembled in a single `/project` dispatch. Both were documented as honest substrate limitations (not faked) and worked around via per-aggregate reads + the deterministic command factory.

4. **Two live proofs only run with infrastructure that the sandbox lacks.** The command-pipeline lane (4.5) and the reminder-recovery lane (4.6) both `Assert.Skip` when Redis :6379 / Dapr placement :50005 / scheduler :50006 are absent. The deterministic adapter tests cover the *decision logic*, but the end-to-end persist-then-publish and restart→reissue round-trips have not been exercised in CI — only locally with `dapr init`. This is the Epic 3 `dotnet test` friction's larger cousin: the runtime stories are inherently infrastructure-bound.

5. **`IProjectionChangeNotifier` is inert in the DomainService SDK.** It is registered only in `Hexalith.EventStore.Server`, so `GetService<IProjectionChangeNotifier>()` is always `null` in the Works host and the notify-on-change call never fires live. The change-detection *decision* (did this delivery change a tenant's what's-next set/order?) is computed and the call is null-tolerant, but live SignalR-style notification is still unwired (deferred, documented in 4.5 DC3).

6. **Submodule pointer drift carried from Epic 3 and grew.** Epic 3 retro's D6 (FrontComposer/Parties gitlink drift) was never cleaned. At Epic 4 retro time `git status` shows **six** modified submodules (`Hexalith.Builds`, `.Commons`, `.EventStore`, `.FrontComposer`, `.PolymorphicSerializations`, `.Tenants`). Some of this is now *intentional* — 4.5 deliberately aligned to the checked-out EventStore submodule — but it has never been committed/resolved, so it appears in every story's working tree.

## 5. Key Insights

- **The pure-core / deferred-runtime bet paid off end-to-end.** Three epics of "build the pure decision, name the deferred seam" converged in Story 4.6 with the kernel still clock-free and the catalog still 36. The reactor translators written in Epic 3 were dispatched in Epic 4 unchanged. Good invariants (terminal no-op idempotency, deterministic reminder names) meant recovery needed *no* new dedup machinery — duplicate resumes and duplicate terminal commands are simply no-ops at `Handle`.
- **Idempotency designed once is the load-bearing pattern for recovery.** "At-least-once dispatch + idempotent target commands + checkpoint-after-attempt" is the whole recovery story. Because the aggregate already no-ops a second resume/terminal, replay-after-crash is *safe by construction* rather than by a recovery-specific mechanism.
- **Honesty about substrate limits is a feature.** Twice the team hit an EventStore gateway boundary and chose to document it and prove the decision logic deterministically rather than fake a tenant-wide scan. The retro endorses this: a documented limitation with deterministic coverage beats a green-but-hollow integration test.
- **The biggest residual risk is exactly where coverage is thinnest.** The two highest-value end-to-end behaviors (live persist-then-publish, live restart→reissue) are the two that skip in CI. The deterministic lanes are strong, but the live lanes need to actually run somewhere before this is called production-ready.

## 8. Next Steps — No Epic 5; Roadmap-Level Preparation

Epic 4 closes the defined roadmap (Epics 1–4). There is **no Epic 5 in `epics.md`**, so there is no next-epic dependency analysis to run. The meaningful "what's next" is two-fold:

1. **Make the runtime proof real (highest value).** Stand up an infrastructure-equipped lane (Docker + `dapr init` + placement/scheduler) so the two Tier-3 lanes (E4) actually execute, converting the deterministic-only proofs of persist-then-publish and restart→reissue into observed end-to-end runs.
2. **Theme 3–6 productization (post-v1, explicitly out of scope so far).** UI / web shell / SignalR surface, executor routing & eligibility (`IExecutorRouter` impl), `AuthorityLevel` enforcement, MCP/chatbot/email adapters, cost governance. These were deferred by design across all four epics and would be the substance of any future roadmap.

If a future epic is planned, the standing inputs it should account for are E1–E3 (substrate read-surface limits) and the live-lane gap (E4) — none of which contradicts existing plans; they are forward constraints, not defects.
